Introduction: How My Understanding of AI Changed

My academic background before this course was in Remote Sensing and GIS — comfortable with satellite imagery, spatial databases, and desktop GIS platforms, but lacking any systematic understanding of how machine learning could reason about spatial patterns, synthesise heterogeneous evidence, or answer natural-language questions about geographic phenomena. I came in seeing AI primarily as a black box: an input–output machine. The Agentic RAG project changed that picture entirely.

For the first time, I was confronted with the reality that building a functional AI system is not a modelling problem — it is, at its core, an integration problem. Perhaps the shift that surprised me most was discovering the centrality of language in AI system design. The most consequential design decisions were not numerical; they were linguistic.

AI Concept-to-Application Map

12 concepts applied in the Agentic RAG project
1

Classical ML: Clustering (DBSCAN)

Session 3

Spatial clustering of GPS trajectory points to identify tourist dwell zones. ε = 0.002°, min_samples = 5. [10]

Algorithm selection is a domain decision, not just a technical one. DBSCAN's noise-point handling is essential for GPS data where outliers represent transit, not tourism.
2

Data Quality & Filtering

Session 4

Speed-filter discards GPS samples implying >150 km/h; language & place-relevance filters clean YouTube comments.

Data quality gates are not preprocessing — they are the first model. I began treating data filtering as a deliberate design decision, not a chore.
3

Feature Engineering

Session 5

Composite GPS density score: weighted blend of DBSCAN cluster ratio (w=0.6) and base density (w=0.4). Confidence-floor feature.

Feature design is where domain knowledge enters the system. Choosing 0.6/0.4 was an epistemological decision expressing my belief that empirical cluster data is more reliable than calibrated priors.
4

Transformer Architecture / Attention

Session 6

XLM-RoBERTa for multilingual sentiment [9]; MiniLM for 384-d dense sentence embedding [7].

Transformers contextualise each token relative to every other token. 'Crowded' meaning different things at a heritage site vs. a market was handled automatically — without any rule I wrote.
5

Transfer Learning & Fine-tuning

Session 6

Both models applied zero-shot to the tourism domain, leveraging multilingual pretraining.

Transfer learning is not a shortcut — it is a form of prior knowledge. Using a multilingual model allowed Thai–English mixed comments to be processed coherently.
6

Vector Embeddings & Similarity Search

Session 7

384-d L2-normalised dense vectors; FAISS IndexFlatIP for exact cosine similarity search. [7][8]

L2 normalisation converting inner product to cosine distance was a small insight with large consequences: without it, long documents would dominate retrieval regardless of semantic relevance.
7

Retrieval-Augmented Generation (RAG)

Session 8

Semantic query encoding → top-k FAISS search → context assembly → LLM generation conditioned on retrieved passages. [4][6]

RAG changed my understanding of hallucination. It is not a model failing — it is a model generating without evidence. RAG provides the evidence.
8

Agentic AI & ReAct

Session 9

LLM agent autonomously decides when to invoke rag_retrieve(), get_hotspot(), or get_sentiment_summary() based on reasoning traces. [5]

Tool descriptions are as important as tool implementations. When I wrote vague descriptions, the agent invoked the wrong tools. When I rewrote them precisely, it routed correctly.
9

Prompt Engineering

Session 9

System prompt encodes data provenance labels, temporal reasoning guidance, and tool-use policies.

Prompt engineering is not formatting — it is a form of knowledge representation. This reframed prompting as a design discipline.
10

Geospatial AI / GIS Integration

Sessions 2, 8

POI coordinates feed into GPS simulation [2], DBSCAN [10], Folium rendering, and the get_hotspot() tool. GeoJSON bridges analytical and visualisation layers. [3]

Spatial context changes query semantics entirely. 'Is Wat Doi Suthep crowded?' requires GPS data; 'Is it worth visiting?' requires sentiment. The agent needs to know this distinction.
11

Evaluation Metrics (IR)

Session 10

HR@1/3/5 and MRR computed over 38 queries; breakdown by category (crowd, sentiment, planning, overtourism). [11][12]

Per-category evaluation revealed something aggregate metrics hid: the system performed poorly on cross-POI crowd comparisons. Evaluation is not confirmation — it is diagnosis.
12

Containerisation & MLOps

Session 11

Docker + docker-compose orchestrates Streamlit app, mounts data volumes, caches HuggingFace model weights via named volumes.

Model caching exposed a concern I had never considered: operational cost. Without the named volume, each restart re-downloads 1.3 GB. Deployment means 'making it run sustainably'.

Agentic RAG vs. Plain RAG — Systematic Comparison Based on project evaluation results [4][5]
Dimension Plain RAG Agentic RAG
Query handling Single retrieval step — one vector search per query Multi-step: LLM reasons, selects tool, interprets result, iterates
Multi-hop queries Fails — e.g. "Which temple avoids crowds and has historical significance?" returns incomplete answers Handles by decomposing into sequential tool calls (get_hotspot + rag_retrieve)
Data source routing Unified FAISS index only — cannot distinguish GPS vs. sentiment signals Separate tools: rag_retrieve(), get_hotspot(), get_sentiment_summary()
Token overhead Baseline (1×) +62% tokens (reasoning traces + tool calls + result formatting)
Transparency Black-box retrieval; no intermediate reasoning visible ReAct traces expose reasoning chain; data provenance labelled per response
Failure mode Incomplete / hallucinated answers when evidence spans multiple data types Tool misdirection (fixable via precise tool descriptions + negative examples) [5]
Deployment cost Lower — simpler pipeline, fewer API calls Higher — not yet suitable for high-volume production without caching strategy
Overall verdict Agentic RAG is architecturally necessary for multi-source, multi-hop queries — but the 62% token overhead remains an open engineering problem for production scale. Plain RAG remains preferred for simple, single-source lookups.

Narrative Reflection: Formative Insights

3.1 — The Complementarity of GPS and Text Modalities

GPS trajectory data and social media text are not redundant — they are genuinely complementary. GPS density tells me how many people are present and where they concentrate; YouTube comments tell me how those people felt about being there. This insight shaped the agentic tool architecture: separate tools for get_hotspot() (quantitative spatial density) and get_sentiment_summary() (qualitative visitor perspective), rather than a single retrieval tool.

3.2 — Data Provenance as an Ethical Responsibility

The GPS trajectories are simulated, not collected from real tourist devices. The sentiment data represents a non-random sample of English-speaking, YouTube-engaged visitors — systematically underrepresenting domestic Thai visitors. This led me to encode provenance transparency directly into the system prompt. Responsible AI practice is not a documentation checkbox; it is an architectural decision built in from the very start.

3.3 — From Rule-Based Thinking to Learned Filtering

My first instinct was to build a rule-based spam filter using a fixed keyword list. Testing revealed high precision but low recall. The final design combines scored keyword matching with a confidence-floor mechanism on the downstream sentiment model. Domain knowledge is most effectively encoded as structural constraints and scoring weights — not as exhaustive enumeration of rule exceptions. A smooth scoring boundary generalises better than a hard boolean filter.

3.4 — Agentic Failure Modes and the Limits of My Mental Model

The agent occasionally invoked get_hotspot() for sentiment queries, or attempted to retrieve GPS density data for historical questions about temple founding dates. These were not retrieval errors — they were prompt-level issues caused by my own imprecise tool descriptions. Agentic systems require explicit, precise tool-use policies, including negative examples and invocation preconditions, written directly into the system prompt. The agent's behaviour is a function of my prompts as much as its training.

Personal Learning Trajectory

Three Conceptual Shifts During Implementation

Shift 1

From retrieval to generation. I initially believed retrieval alone would suffice for answering tourism queries. The first time the system produced a coherent, context-aware response by combining retrieved passages with language model generation, I understood why the conversational interface is qualitatively superior to a ranked list.

Shift 2

From RAG to Agentic RAG. Plain RAG failed on multi-hop queries such as 'Which temple should I visit first if I want to avoid crowds and prefer historical significance?' No single retrieved document could answer this. Implementing the agentic layer was not an optional enhancement — it was a necessary response to a real failure mode.

Shift 3

From aggregate to diagnostic evaluation. Early evaluation showed acceptable aggregate Hit Rate. Per-category breakdown revealed the system performed substantially worse on cross-POI crowd comparisons. Evaluation serves to find what is wrong, not to confirm what is right.

Limitations & Open Questions

Confidence threshold calibration: A threshold of 0.70 may be appropriate for temples (polarised sentiment) but suboptimal for markets (mixed sentiment). No principled method for setting category-specific thresholds yet.

CRW simulation validation: GPS trajectories are simulated, not collected. Without real tourist device data, the simulation's distributional assumptions cannot be validated against actual visitor behaviour.

Agentic token overhead at scale: The agentic configuration incurs a 62% token overhead relative to plain RAG. No caching or pre-computation strategy yet designed for production query volumes.

Multilingual evaluation: The evaluation set of 38 queries is English-only. Performance on Thai-language queries — a large share of the target user population — remains unknown.

Conclusion

Data Quality is the First Model

Without quality gates, every downstream component degrades. Data filtering is design, not chore.

Language is a Design Medium

Prompts, tool descriptions, and provenance labels are primary design artefacts in modern AI systems.

Evaluation is Diagnostic

Aggregate metrics confirm; per-category breakdowns diagnose. Evaluation serves to find what is wrong.

I came into this course knowing how to use AI tools; I leave it understanding how to think about AI systems. Using a tool means knowing its interface. Understanding a system means being able to reason about its failure modes, its data assumptions, its evaluation blind spots, and its ethical responsibilities — and being able to make deliberate design choices that address each of these.

Course Feedback

The most valuable aspect of this course was the decision to anchor learning in a single, end-to-end project rather than a series of disconnected exercises. Working through the full stack — from raw data filtering all the way to containerised deployment — forced me to understand how concepts interact under real constraints. The introduction of the ReAct agentic paradigm was the most intellectually generative part of the course; I would welcome even earlier exposure to agentic failure modes. If I were to suggest one addition, it would be a structured peer-review session at the retrieval evaluation stage, where observing how other teams approached chunking and query design would have accelerated my own diagnostic thinking considerably.

Glossary of Technical Terms

CRW — Correlated Random Walk
A stochastic movement model in which each step direction is partially correlated with the previous step. Used to simulate realistic tourist GPS trajectories.
DBSCAN
Density-Based Spatial Clustering of Applications with Noise. Groups points by local density rather than distance to a centroid; designates low-density points as noise.
FAISS
Facebook AI Similarity Search. An open-source library for efficient similarity search over large collections of dense vector embeddings.
GPS — Global Positioning System
A satellite-based navigation system providing geographic coordinates. GPS trajectory data records sequences of timestamped coordinates to infer movement patterns.
HR@k — Hit Rate at k
An IR metric measuring the proportion of queries for which the correct answer appears within the top-k ranked results.
LLM — Large Language Model
A deep learning model trained on large text corpora to understand and generate human language. In this project, serves as the orchestration layer.
MiniLM
A compact sentence transformer model producing dense vector embeddings optimised for semantic similarity tasks, generating 384-dimensional embeddings.
MLOps — Machine Learning Operations
Practices combining ML, software engineering, and operations to deploy, monitor, and maintain ML models reliably in production environments.
MRR — Mean Reciprocal Rank
An IR metric averaging the reciprocal of the rank at which the first correct result appears across a set of queries.
NLP — Natural Language Processing
A subfield of AI focused on enabling computers to understand, interpret, and generate human language.
POI — Point of Interest
A specific geographic location relevant for analysis or tourism planning. In this project: temples, markets, and natural attractions in Chiang Mai.
RAG — Retrieval-Augmented Generation
An AI architecture that augments LLM generation by first retrieving relevant documents and conditioning output on retrieved evidence. Reduces hallucination.
ReAct — Reasoning and Acting
An agentic AI framework (Yao et al., 2023) in which a language model alternates between generating explicit reasoning traces and invoking external tools.
XLM-RoBERTa
Cross-lingual Language Model RoBERTa. A multilingual transformer pre-trained on text in 100 languages. Used here for zero-shot sentiment classification of mixed Thai-English reviews.
